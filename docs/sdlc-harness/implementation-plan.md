# BitFun 可配置开发体验与工程治理实施计划

> 范围：基于 [design.md](design.md)，定义 BitFun 面向外部目标项目落地快速路径、配置化策略、安全边界、团队治理和复杂生命周期能力的阶段路线。
> 目标：先证明默认开发体验简洁快速，再逐步增加上下文信心、团队规则、证据链、图谱、评估和强治理能力。

## 1. 执行定位

本计划采用“先快、后配、按风险显露”的落地路线：

```text
架构跑道
  -> 快速路径
  -> 安全边界
  -> 上下文保障
  -> 团队治理
  -> 生命周期上下文
  -> 评测与优化
```

每个阶段都必须回答两个问题：

1. 这个能力是否让普通开发更快、更清楚、更少中断。
2. 这个能力如何在复杂项目或高风险场景中提供更强治理。

## 2. 执行原则

| 原则 | 执行含义 |
|---|---|
| 快速路径先行 | P0 以用户完成有用任务为核心，不以门禁结果为核心 |
| 安全边界独立 | prompt 注入、网络、凭据、hook/MCP、shell、删除、发布先落地 |
| 渐进画像 | 项目画像先识别可运行入口、规则来源和验证能力，完整建模按场景补齐 |
| 提示优先 | 默认建议模式；强制要求/阻断只来自安全、组织策略或用户显式升级 |
| 配置复用 | 先读取 AGENTS.md、CONTRIBUTING、CI、CODEOWNERS、`.github`、`.coderabbit.yaml`、`.gitlab/duo` |
| 证据后台化 | 证据包先做内部契约和 PR/就绪度投影，不作为默认 UI |
| 成本可见 | 每个审查/检查/升级记录耗时、token、用户中断和跳过原因 |
| 反证驱动 | 一旦提示噪音、误升级、误阻断过高，就收缩策略 |

## 3. 阶段路线图

| 阶段 | 主题 | 阶段成果 | 进入下一阶段条件 |
|---|---|---|---|
| P-1 | 架构和产品边界 | 配置化策略结构、安全边界结构、最小事件、配置优先级 | 能明确安全、质量、项目规则、用户覆盖的边界 |
| P0 | 快速路径 + 安全边界 | 快速项目打开、轻量项目理解、低噪音安全提示、简洁任务摘要 | 普通任务能低摩擦完成，安全越界不会静默发生 |
| P1 | 上下文保障 | 风险触发提示、推荐检查、变更就绪度、可选定向审查 | 高风险变更解释升级原因，低风险变更保持快速路径 |
| P2 | 团队治理 | 仓库(repo)/路径(path)/团队配置、审查强度、强制检查、PR 证据包 | 团队规则统一受管工作区体验，临时任务保持低摩擦 |
| P3 | 生命周期上下文 | 交付物图谱、需求/发布/事故链接、风险接受 | 复杂项目能追溯需求到发布和事故回流 |
| P4 | 评测与优化 | 轨迹回放、治理指标、策略 A/B、保留集 | 治理策略能用速度、质量、安全和成本联合评估 |

## 4. P-1：架构和产品边界

| 交付件 | 内容 | 验收方式 |
|---|---|---|
| 配置化策略契约 | 模式、原因、展示层级、检查、审查模式、覆盖选项 | 能解释每次提示/升级/阻断来源 |
| 用户画像映射 | 快速试验者、独立开发者、团队审查人、架构负责人、安全合规、平台维护者的默认入口 | 能说明同一能力对不同人群的展示和升级方式 |
| 安全边界契约 | 权限、沙箱、网络、凭据、主动配置信任、应急放行 | 安全决策不依赖门禁或质量策略 |
| 配置优先级 | 组织拒绝策略/受管强制要求、安全、已确认路径/团队规则、工作区配置、任务覆盖、用户默认值 | 冲突规则有确定优先级，用户覆盖不能绕过强策略 |
| 最小事件注册表 | `project.opened`、`task.started`、`policy.decided`、`security.decided`、`verification.completed` | 快速路径和安全提示可追踪 |
| 证据展示层级 | none、摘要、evidence_refs、full_pack | 证据包按 PR、团队、发布和复盘场景显露 |
| 产品指标规格 | 首次有用动作耗时、打断率、误升级率、应急放行率 | 阶段验收不只看质量 |

退出条件：可以描述 P0 所需对象和事件，且每个高权限动作都有 allow/ask/deny/应急放行的明确路径。

## 5. P0：快速路径 + 安全边界

### 5.1 阶段目标

让用户打开任意外部项目后，能快速完成一次普通开发任务，并在不理解内部治理术语的情况下获得简洁结果和必要安全保护。

### 5.2 交付件

| 交付件 | 内容 |
|---|---|
| 轻量项目理解 | 识别语言、包管理器、常用脚本、git 状态、README/AGENTS/CONTRIBUTING/CI 入口 |
| 快速任务摘要 | 展示改动、运行命令、未验证项、下一步建议 |
| 安全边界 v0 | 工作区写、shell、网络、凭据、删除、跨根目录写入的基础决策 |
| 主动配置发现 | 发现 hook、plugin、MCP、自定义工具和智能体规则，默认未信任 |
| 单次应急放行 | 单次命令、域名、目录、会话范围的临时放行 |
| 最小质量数据面（QDP） | 记录任务、工具、验证、安全决策和用户覆盖 |

### 5.3 验收成果

- 无 git 临时目录可以完成小工具任务，并获得轻量结果摘要。
- 常规项目可以在不配置 `.bitfun` 的情况下识别主要开发命令。
- 安全敏感动作会提示原因、范围和可选隔离路径。
- 用户可以一次性放行低/中风险越界，但不能静默持久化。
- 任务结束时有简洁信心摘要。

### 5.4 过程风险

| 风险 | 处置 |
|---|---|
| 安全提示太频繁 | 先用沙箱/白名单降噪，再提示 |
| 项目理解不准 | 只把推断结论用于提示，不用于强策略 |
| 用户跳过太多 | 显示残余风险、覆盖范围和恢复建议 |
| 临时项目被误判成团队项目 | 无显式团队配置时保持 `fast` |

## 6. P1：上下文保障

### 6.1 阶段目标

当任务出现风险或用户准备 PR 时，BitFun 提供上下文信心：为什么这个变更值得额外检查、建议跑什么、跳过会有什么后果。

### 6.2 交付件

| 交付件 | 内容 |
|---|---|
| 配置化策略决策 | `fast/assist/review/guarded/regulated` 运行态和原因 |
| 风险提示 v0 | 路径、模块、操作、历史信号和安全敏感标签 |
| 推荐检查 | 推荐验证命令、替代验证、不可运行原因 |
| 变更就绪度摘要 | PR 前摘要：变更、验证、风险、未覆盖项 |
| 定向审查触发 | 仅在高风险或证据薄弱的中风险场景建议定向审查 |
| 证据包摘要模式 | 只生成摘要和证据引用，不默认完整证据包 |

### 6.3 验收成果

- 低风险改动不触发深度审查。
- 高风险改动能解释升级原因。
- 检查建议有触发原因和取消条件。
- CI/private env 不可运行时输出替代建议、信心影响和后续补证路径。
- PR 摘要减少审查人追问，但不默认阻塞。

## 7. P2：团队治理

### 7.1 阶段目标

让团队通过配置文件和现有规则资产统一 BitFun 行为，并减少每个用户重复手工设置。

### 7.2 交付件

| 交付件 | 内容 |
|---|---|
| `.bitfun/quality.yaml` 或 `bitfun.toml` | 模式默认值、路径规则、强制检查、审查强度、安全策略 |
| 现有规则导入 | AGENTS.md、CONTRIBUTING、CODEOWNERS、CI、`.github/instructions`、`.coderabbit.yaml`、`.gitlab/duo` |
| 路径级策略 | 核心模块、安全目录、docs、tests、generated code 分别配置 |
| PR 证据包投影 | 团队/守护场景生成可追踪证据引用 |
| 强制检查模式 | 只在配置或确定性风险下强制要求 |
| 风险接受 | 操作者、原因、范围、残余风险、过期时间 |

### 7.3 验收成果

- 团队能用 repo 配置统一体验。
- 个人临时任务使用个人默认值；受管控工作区应用团队配置。
- 路径规则冲突能显示并要求确认。
- 强制要求/阻断有明确来源，不由模型单独触发。

## 8. P3：生命周期上下文

### 8.1 阶段目标

为复杂项目提供需求、PR、发布、事故、回归资产之间的追溯能力，但只在用户需要解释、发布或复盘时显性化。

### 8.2 交付件

| 交付件 | 内容 |
|---|---|
| 交付物图谱最小闭环 | `diff -> verification -> evidence_pack -> PR` |
| 需求影响候选 | 高风险需求/API/设计变更影响候选和人工确认 |
| 发布就绪度 | CI、审查、已知风险、回滚、遥测汇总 |
| 事故到测试回归 | 事故回溯发布、PR、diff 和测试缺口 |
| 过期证据 | 新提交、策略、风险或审查范围变化后失效 |

### 8.3 验收成果

- 普通任务不需要看到图谱。
- 复杂 PR 可以解释证据和关系来源。
- 发布就绪度能说明可发布、不可发布或已接受风险。
- 事故能转成回归候选或规则更新。

## 9. P4：评测与优化

### 9.1 阶段目标

用真实任务和产品指标优化智能体行为、治理策略和质量治理，避免只优化基准测试或只优化强管控。

### 9.2 交付件

| 交付件 | 内容 |
|---|---|
| 轨迹回放 | 回放任务、命令、验证、提示和覆盖 |
| 评测卡 | 任务集、判定标准、模型/上下文/工具/策略版本、成本、安全事件 |
| 治理策略 A/B | 比较提示频率、误升级、成功率、用户放行和质量结果 |
| 保留集 | 防泄漏任务集和真实项目回归 |
| 策略校准 | 合入后缺陷、审查阻塞项、CI 失败和用户反馈回流 |

### 9.3 验收成果

- 能证明新策略没有让普通任务变慢。
- 能证明安全提示减少盲目确认。
- 能证明高风险场景质量更稳定。
- 能发现哪些门禁、审查或证据能力没有产品价值，应下线或降级。

## 10. 看护指标

详细口径见 [metrics-spec.md](governance/metrics-spec.md)。P0/P1 必须优先看这些指标：

- 首次有用动作耗时。
- 用户打断率。
- 安全提示接受/忽略率。
- 应急放行率和范围。
- 误升级率。
- 低风险任务完成率。
- 推荐检查执行率。
- PR 就绪度采用率。
- 强制检查精度。
- 合入后缺陷 / 审查阻塞项反馈。

## 11. 发布和 PR 策略

- P0/P1 默认不创建强制 GitHub 检查；优先本地摘要和 PR 文本块。
- 团队治理开启后，才将就绪度或门禁投影到 PR 状态/评论。
- 阻断只允许来自安全边界、组织策略、确定性失败或明确强制要求策略。
- 每次发布 PR 文档必须说明默认体验是否变重，以及如何避免普通项目被重流程影响。

## 12. 关键里程碑

| 里程碑 | 判定标准 |
|---|---|
| M0：边界清晰 | 配置化策略、安全边界、证据展示层级和配置优先级稳定 |
| M1：快速路径可用 | 普通项目无需配置即可完成有用开发任务 |
| M2：安全低噪音 | 高风险动作可控，普通命令不频繁打断 |
| M3：上下文信心可解释 | 高风险变更有理由、建议检查和跳过后果 |
| M4：团队规则可复用 | 仓库/路径/团队配置能统一 PR 就绪度和强制检查 |
| M5：复杂项目可追溯 | 证据包/图谱支撑 PR/发布/事故，快速路径保留轻量摘要 |
| M6：策略可评估 | 治理策略能用效率、质量、安全和成本联合衡量 |

## 13. 完成标准

这组能力完成时，BitFun 应表现为：

- 普通开发像轻快助手。
- 高风险动作像可靠安全执行环境。
- 团队项目像可配置协作系统。
- 复杂项目像可追溯工程平台。
- 所有复杂技术都服务于明确体验：更快完成任务、更准识别风险、更清楚解释信心或更可靠复盘。
