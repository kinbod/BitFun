# BitFun 可配置开发体验与工程治理设计

> 范围：定义 BitFun 加载外部软件工程后的默认产品体验、复杂项目来源、配置化策略模型、执行安全边界、模块职责和长期演进路径。
> 边界：本文不以 BitFun 仓库自身治理为主线。BitFun 自身的文档、模块边界或工程治理问题应作为独立内部验证与平台建设输入承载，不混入产品目标架构。

## 1. 设计目的

BitFun 的目标是成为面向任意目标项目的可配置智能体开发产品：默认帮助用户快速理解、修改、运行、验证和交付；当任务、权限、发布或团队流程变复杂时，通过项目、路径、团队和用户配置逐步显露上下文、证据、审查、策略和治理能力。

主设计回答四个问题：

1. 如何让普通项目和临时小工具默认保持低摩擦。
2. 如何在复杂项目、团队协作、发布和合规场景中逐步升级治理强度。
3. 如何让执行安全独立生效，覆盖 prompt 注入、凭据、网络、hook、MCP 和发布等高风险动作。
4. 如何让证据包、交付物图谱、质量数据面（QDP）、风险分类和门禁等技术服务于用户任务，并在 PR、团队、发布或事故场景按需显露。

一句话目标：

```text
默认快速开发，风险场景按配置升级治理，安全边界始终生效。
```

## 2. 市场与产品判断

当前先进开发工具的共同趋势是“快速执行 + 项目规则 + 可选审查 + 安全边界”：

| 市场信号 | 对 BitFun 的启发 |
|---|---|
| [GitHub Copilot 自定义指令](https://docs.github.com/en/copilot/how-tos/copilot-on-github/customize-copilot/add-custom-instructions/add-repository-instructions) 支持 repo、path 和 AGENTS.md 指令 | 项目知识应优先读取现有规则文件，并按路径/上下文渐进适用 |
| [GitHub Copilot 代码审查](https://docs.github.com/en/copilot/how-tos/use-copilot-agents/request-a-code-review/use-code-review) 默认评论，不计入强制要求审批 | AI 审查优先提供低摩擦提示，阻塞能力由项目或组织策略配置 |
| [Codex 沙箱/审批](https://developers.openai.com/codex/agent-approvals-security) 和 [Claude 权限](https://code.claude.com/docs/en/permissions) 拆分沙箱与审批 | 执行安全必须独立于质量治理，且用技术边界减少弹窗 |
| [CodeRabbit 审查强度](https://docs.coderabbit.ai/reference/configuration) 默认 `chill`，`assertive` 可选 | 审查强度应可调，默认减少噪音 |
| [CodeRabbit 路径指令](https://docs.coderabbit.ai/configuration/path-instructions) 和 [GitLab Duo 指令](https://docs.gitlab.com/user/gitlab_duo/customize_duo/review_instructions/) 支持路径规则 | 复杂项目来源主要来自路径、团队和场景，不只是项目整体 |
| [GitLab 警告模式](https://docs.gitlab.com/user/application_security/policies/merge_request_approval_policies/) 用于先验证策略影响 | 强策略应先建议/警告，再进入强制/阻断 |
| [Kiro steering](https://kiro.dev/docs/steering/) 区分工作区/全局/团队和 inclusion 模式 | 项目规则需要作用域、优先级和加载时机 |
| [Jules](https://jules.google/) 强调异步任务、计划、diff 和 PR 审查 | 长任务可异步代理，用户界面应围绕计划、diff 和批准组织 |

这些信号说明：BitFun 的差异化应落在“可配置治理强度”和“安全边界常驻”上，让不同复杂度、不同风险和不同团队规则都能获得合适的默认体验。

## 3. 产品体验原则

| 原则 | 设计含义 |
|---|---|
| 默认快速 | 普通任务使用轻量项目理解、简洁摘要和低成本验证建议 |
| 渐进显露 | 风险、团队配置、PR/发布阶段或用户选择触发更多治理信息 |
| 安全独立 | prompt 注入、hook/MCP/网络/凭据/shell/删除/发布统一走安全边界 |
| 解释优先 | 每一次提示、验证、确认或阻断都必须解释来源和后果 |
| 用户可决策 | 允许单次、会话或 worktree 范围的临时放行，但有范围、期限和残余风险 |
| 项目规则复用 | 优先读取 AGENTS.md、CONTRIBUTING、CI、CODEOWNERS、`.github`、`.coderabbit.yaml`、`.gitlab/duo` 等已有资产 |
| 复杂项目友好 | 单体多项目仓库、多仓库、私有 CI、不稳定检查、合规审计、灰度发布和事故回流都有升级路径 |
| 技术后台化 | 图谱、质量数据面、证据包和评测优先服务摘要、解释、审查和复盘 |

## 4. 用户画像与体验策略

| 用户画像 | 默认体验 | 升级触发 | 关键保护 |
|---|---|---|---|
| 快速试验者 | `fast`，无配置启动，显示任务结果和下一步建议 | 高权限 shell、网络、凭据、删除、跨目录写或准备分享 | 安全边界常驻，质量建议可跳过 |
| 独立开发者 | `assist`，复用 README、脚本和仓库规则，给推荐检查 | 准备 PR、跨模块改动、验证缺口 | 未验证项和信心摘要清晰可见 |
| 团队开发者 / 审查人 | `review`，生成 PR 就绪度、风险标签和审查建议 | 受保护分支、CODEOWNERS、路径规则、团队配置 | 审查强度可配置，阻断来源可追踪 |
| 技术负责人 / 架构师 | `guarded`，关注核心路径、接口、迁移和发布影响 | API/数据/权限/架构边界变化 | 影响候选、证据引用和风险接受分离 |
| 安全 / 合规负责人 | `regulated`，保留审计、批准、发布和回滚证据 | 凭据、网络、MCP、hook、生产发布、受管项目 | 组织策略高于本地覆盖 |
| 平台 / 插件维护者 | 按任务进入 `assist` 或 `guarded`，重点管理主动配置 | hook、plugin、MCP、自定义工具或远程执行面变化 | 主动配置先作为未授权候选，隔离后授权 |

用户画像用于选择默认解释方式、提示密度和建议入口；最终运行态由任务意图、路径风险、执行动作、团队策略和安全边界共同决定。快速路径界面使用“任务结果、信心摘要、安全确认、下一步建议”等产品语言；证据包、图谱、门禁和审计作为 PR、团队、发布、事故或监管场景的显式能力。

## 5. 目标用户路径

### 5.1 快速路径：默认路径

```text
打开项目或目录
  -> 轻量理解项目
  -> 用户提出任务
  -> 修改 / 运行 / 检查
  -> 输出简洁结果和信心摘要
  -> 可选生成 PR 摘要或下一步检查建议
```

快速路径适合：

- 无 git 临时目录。
- 演示、小工具、脚本、文档、探索性改动。
- 个人项目或低风险仓库。
- 用户明确只想快速试验。

快速路径的输出以任务结果、已运行检查、未验证项和下一步建议为主；门禁、审计、风险接受、交付物图谱和完整证据包仅在 PR、团队治理、发布或事故复盘场景显露。

### 5.2 上下文保障：风险出现时升级

```text
变更或动作出现风险
  -> 解释触发原因
  -> 建议检查、沙箱、审查人或上下文补充
  -> 除非团队策略或关键安全风险要求，否则保持建议模式
```

触发例子：

- 改到核心路径、权限、网络、AI 适配器、数据迁移、发布配置。
- 项目规则冲突或关键验证不可运行。
- 用户准备生成 PR。
- 证据过期或上次审查不再覆盖新 diff。

### 5.3 团队治理：项目或组织要求时显露

```text
项目或团队配置启用
  -> 应用路径规则和审查强度
  -> 生成变更就绪度或证据包
  -> 执行强制检查、定向审查或风险接受
  -> 投影到 PR 或发布流程
```

团队治理由显式配置、受保护分支、组织策略、合规要求或用户选择触发，并通过路径规则、审查强度、强制检查和风险接受统一团队体验。

### 5.4 安全边界：始终启用

```text
工具或配置请求能力
  -> 分类安全风险
  -> 在沙箱中允许 / 询问 / 拒绝 / 应急放行
  -> 记录带范围的决策
```

安全边界只关注执行安全，不判断质量好坏。详见 [security-boundary.md](architecture/security-boundary.md)。

## 6. 复杂项目来源

BitFun 面对的复杂性来自多种来源叠加：

| 复杂来源 | 例子 | 产品要求 |
|---|---|---|
| 结构复杂 | 单体多项目仓库、多语言、多服务、生成代码、跨仓库依赖 | 渐进画像和路径级规则 |
| 流程复杂 | issue/spec/PR/发布/事故分散在多个系统 | 交付物图谱后台关联，按需显露 |
| 验证复杂 | 私有依赖、不稳定 CI、本地环境不完整、测试分层不清 | 替代验证建议和不可验证解释 |
| 风险复杂 | 权限、网络、凭据、迁移、发布、远程工作区 | 安全边界 + 守护模式 |
| 团队复杂 | 负责人、CODEOWNERS、组织策略、合规审计 | 团队治理和风险接受 |
| 时间复杂 | 热修复、临时修复、长程异步任务、多智能体并行 | 任务级模式和工作区隔离 |
| 信息复杂 | 旧文档、冲突规则、prompt 注入、未知来源配置 | 来源、信任、冲突和新鲜度 |

因此，配置化策略以项目配置作为默认值，同时结合当前任务、目标路径、执行动作和用户覆盖决定运行态。

## 7. 顶层领域模型

| 领域 | 核心对象 | 体验作用 |
|---|---|---|
| 项目理解 | 工作区、仓库、语言、框架、模块、规则来源、验证能力 | 让用户少解释，减少智能体走错路 |
| 用户画像 | 用户类型、任务熟悉度、偏好、授权范围、受管状态 | 决定解释方式、提示密度和默认入口 |
| 任务与意图 | 任务、模式、阶段、用户覆盖、会话模式 | 决定当前是快跑、准备 PR、发布还是应急 |
| 执行安全 | 权限、沙箱、网络域名、凭据访问、主动配置信任、应急放行 | 防止 prompt、配置或工具越权 |
| 变更信心 | 变更摘要、验证摘要、风险提示、未关闭问题、跳过检查 | 给用户可理解的信心和下一步行动 |
| 团队治理 | 路径规则、审查强度、强制检查、风险接受、审计策略 | 在团队/高可靠场景统一体验 |
| 生命周期上下文 | issue、spec、计划、PR、发布、事故、学习资产 | 支撑复杂项目追溯和复盘 |
| 评测与学习 | 轨迹回放、评测任务、判定标准、反馈、指标 | 证明策略真的改善体验和质量 |

## 8. 逻辑架构

```text
用户界面
  快速工作台 / 变更摘要 / PR 就绪度 / 发布审查 / 设置

配置化策略面
  意图识别 / 策略画像 / 风险提示 / 审查画像 / 覆盖策略

安全边界
  沙箱 / 权限 / 网络 / 凭据 / 主动配置信任 / 应急放行

交付物与证据面
  证据包 / 交付物图谱 / 问题生命周期 / 风险接受

质量数据面
  生命周期事件 / 执行轨迹 / 验证摘要 / 指标 / 本地审计

项目集成面
  Git / CI / Issue / 文档 / 发布 / 观测 / 知识库

智能体与工具运行时
  会话 / 智能体 / 终端 / 文件系统 / MCP-LSP / Hooks / 适配器
```

关键边界：

- 用户界面展示当前模式下必要信息，并把策略选择交给配置化策略面。
- 配置化策略面决定体验强度、检查建议、证据展示层级和用户覆盖选项。
- 安全边界负责权限、沙箱、网络、凭据和高风险动作隔离。
- 交付物、证据和质量数据面支撑解释、审查、发布和复盘。
- 项目集成面适配外部系统，并把外部语义映射为内部稳定事件和契约。
- 智能体运行时执行任务，并通过策略面和安全边界获取质量结论与授权状态。

## 10. 配置层级

BitFun 应复用现有生态配置，并提供自己的最小补充：

| 层级 | 例子 | 作用 |
|---|---|---|
| 用户偏好 | 用户选择快/严、语言、默认授权范围 | 个体体验默认值 |
| Session/任务覆盖 | “这次快速放行网络”“本任务只读分析” | 临时治理强度 |
| 工作区配置 | `.bitfun/config.toml` 或 `.bitfun/quality.yaml` | BitFun 专属模式、检查、安全策略 |
| 现有仓库规则 | AGENTS.md、CONTRIBUTING、CODEOWNERS、CI、`.github/instructions` | 项目知识和规则来源 |
| 工具特定规则 | `.coderabbit.yaml`、`.gitlab/duo/*`、`.kiro/steering/*` | 外部工具和路径级经验 |
| 组织策略 | 受管配置、受保护分支、企业策略 | 强制策略和不可绕过限制 |

优先级原则：

```text
组织拒绝 / 受管强制要求
  > 安全边界
  > 最近确认的路径或团队规则
  > 工作区配置
  > 允许范围内的会话/任务覆盖
  > 全局/用户默认值
```

这条优先级用于冲突裁决。每条规则必须声明执行模式：`default`、`advisory`、`required` 或 `blocking`。用户的任务覆盖可以降低建议模式噪音、跳过推荐检查或请求风险接受；组织拒绝策略、受管控路径规则、强制检查、安全边界和受保护分支规则保持更高优先级。

## 11. 分阶段产品路径

| 阶段 | 主题 | 用户价值 |
|---|---|---|
| P0 | 快速路径 + 安全边界 | 快速完成普通开发任务，同时不放开关键安全风险 |
| P1 | 上下文保障 | 高风险变更出现时给出清晰提示、验证建议和 PR 就绪度 |
| P2 | 团队治理 | 项目/团队可以用配置文件统一规则、审查强度和强制检查 |
| P3 | 生命周期上下文 | 复杂项目把需求、PR、发布、事故和回归资产连起来 |
| P4 | 评测与优化 | 用真实任务、反馈和指标持续优化治理策略和智能体行为 |

## 12. 硬约束

| 约束 | 原因 | 设计要求 |
|---|---|---|
| 门禁按配置显露 | 普通项目和临时任务需要低摩擦 | 门禁在 PR/团队/强策略场景显露 |
| 安全边界独立执行 | prompt 注入、凭据、网络、hook 风险与质量无关 | 安全边界独立强制执行 |
| 技术契约后台化 | 用户以任务、摘要和建议完成工作 | 证据包、质量数据面和图谱后台化 |
| 未知先进入建议态 | 新项目初期必然信息不足 | 先提示和建议，安全或团队策略再阻断 |
| 用户可临时放行 | 小工具和应急场景需要效率 | 应急放行有范围、期限、记录和撤销 |
| 组织策略可强制 | 企业合规需要不可绕过 | 受管策略高于本地覆盖 |
| 模型输出作为候选 | LLM 适合召回、总结和解释 | 确定性证据、用户决策和策略改变状态 |

## 13. 全局风险与治理

| 风险 | 后果 | 治理策略 |
|---|---|---|
| 产品默认过重 | 普通用户不采用，临时任务流失 | P0 以快速路径和低摩擦指标验收 |
| 质量能力被弱化成玩具 | 复杂项目无法信任 | 团队/守护/监管场景保留证据包、门禁、审计 |
| 安全提示太频繁 | 用户养成盲点确认 | 沙箱、白名单、范围和域名策略降噪 |
| 应急放行被滥用 | 安全边界失效 | 范围、期限、组织禁用、隔离建议和审计 |
| 项目规则被注入污染 | 智能体按恶意文档执行 | 规则来源、信任、新鲜度和恶意指令检测 |
| 路径规则冲突 | 同一文件应用互相矛盾策略 | 最近规则 + 冲突展示 + 人工处理 |
| 图谱和证据过早显露 | 用户体验复杂化 | 只在解释、PR、发布、事故时显性化 |
| 指标只看质量不看效率 | 产品优化方向偏向重管控 | 指标同时覆盖首次有用动作耗时、打断、覆盖和误升级 |
| 平均体验掩盖局部人群问题 | 快速用户或受管团队中至少一类体验失衡 | 指标按用户画像、任务风险、治理模式和受管状态切片 |

## 14. 成功状态

- 新用户打开普通项目，可以在几分钟内完成一次有用改动。
- 当任务触及安全敏感动作，BitFun 能低噪音地说明风险并给出安全放行或隔离路径。
- 当用户准备 PR，BitFun 能给出简洁变更就绪度，并按需引用证据。
- 当团队启用规则，BitFun 能稳定执行路径级检查、审查强度、强制检查和风险接受。
- 当项目复杂到跨需求、发布和事故，交付物图谱和证据包能提供追溯。
- 产品指标能同时证明速度、信心、安全和质量。
